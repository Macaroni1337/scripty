## WHAT YOU ARE BUILDING

Build a Python-based CLI tool called **Scripty** — a guided, hand-holding penetration testing companion for people who know *what* they want to do but not exactly *how* to do it. Think: "script kiddie, but make it fashion." The vibe is a cool, charismatic hacker mentor living in your terminal. It wraps industry-standard tools (nmap, netcat, gobuster, hydra, wireshark, sqlmap, etc.) in a rich, interactive CLI experience with menus, prompts, progress indicators, color, ASCII art, and plain-English explanations of what's happening at every step.

This is NOT a raw tool. It is a guided experience. Every action explains itself. Every command shows the raw syntax it used. Every result is parsed and summarised in human language.

## PERSONALITY & AESTHETIC

- Name: **Scripty**
- Mascot concept: a hoodie-wearing terminal cursor with attitude
- Tone: Snarky, encouraging, slightly chaotic, always knowledgeable. Like a senior pentester who thinks you're kinda dumb but secretly wants you to succeed.
- Color palette (terminal):
  - Primary: Bright cyan `#00FFFF` / `\033[96m`
  - Accent: Hot pink / magenta `\033[95m`
  - Success: Lime green `\033[92m`
  - Warning: Amber `\033[93m`
  - Error: Bright red `\033[91m`
  - Muted/info: Dark grey `\033[90m`
  - Background feel: Dark terminal. Everything pops against black.
- All menus use box-drawing characters (─ │ ╭ ╰ ├ └) for a polished panel look
- ASCII art banner on launch (Scripty logo + tagline: *"pentesting for the rest of us"*)
- Spinner/progress animations for any running process (use `rich` or `halo` library)
- Every module opens with a one-liner explaining what it does and why a pentester would use it

---

## TECH STACK

- Language: **Python 3.10+**
- CLI framework: **`rich`** (panels, tables, spinners, markdown, syntax highlighting)
- Prompts/menus: **`questionary`** or **`InquirerPy`** for interactive arrow-key menus
- Argument parsing (for direct CLI flags): **`typer`** or **`click`**
- Process execution: **`subprocess`** with real-time stdout streaming
- Config/state: **`toml`** or JSON stored in `~/.scripty/config.toml`
- Dependency checking: On launch, verify all external tools are installed and warn if missing
- OS support: Primary target **Kali Linux** / Debian-based. Should also work on macOS with Homebrew tools.

## LAUNCH EXPERIENCE

On `scripty` or `python scripty.py`:

1. Clear the terminal
2. Print the ASCII art banner (cyan + magenta gradient feel)

## MODULE SPECIFICATIONS

Each module follows the same UX pattern:
1. Show a panel header with module name + short description
2. Ask the user for required inputs via `questionary` prompts (with validation)
3. Show the exact command that will be run (syntax-highlighted) and ask for confirmation
4. Run it with a live output stream + spinner
5. Parse and summarise the output in plain English
6. Offer follow-up actions based on results

---

### MODULE 1: RECONNAISSANCE

Sub-menu options:

**1.1 Nmap — Basic Port Scan**
- Inputs: target IP/hostname, scan type (Quick / Full / Stealth / Version Detection / Vuln Script / Custom)
- Maps selection to nmap flags: `-T4 -F` / `-T4 -p-` / `-sS` / `-sV -sC` / `--script vuln` / raw input
- Streams output live
- Parses open ports into a Rich table: Port | Protocol | State | Service | Version
- Suggests next steps based on open ports (e.g. "Port 80 open → run Web Recon module")

**1.2 Nmap — OS Detection**
- Flags: `-O --osscan-guess`
- Summarises detected OS

**1.3 DNS Enumeration**
- Uses `dig`, `host`, `nslookup`, `dnsrecon` if available
- Inputs: domain
- Runs A, MX, TXT, NS, AXFR lookups
- Shows results in a clean table

**1.4 WHOIS Lookup**
- Runs `whois` on domain/IP
- Parses registrar, dates, name servers

**1.5 Subdomain Enumeration**
- Uses `gobuster dns` or `ffuf` with a wordlist
- Input: domain, wordlist selection (offer built-in options + custom path)
- Streams discovered subdomains live

**1.6 Ping Sweep / Host Discovery**
- Uses `nmap -sn` on a CIDR range
- Lists live hosts in a table

### MODULE 2: WEB APPLICATION TESTING

**2.1 Directory & File Busting**
- Tool: `gobuster dir` or `ffuf`
- Inputs: URL, wordlist, file extensions (optional), status codes to show
- Streams results live, highlights juicy finds (admin, login, backup, config, .env, .git)

**2.2 Nikto Web Scan**
- Full nikto scan against target URL
- Parses findings and categorises by severity

**2.3 WhatWeb / Tech Fingerprinting**
- Runs `whatweb` on target
- Shows detected tech stack: CMS, frameworks, server, JS libs

**2.4 SQL Injection — sqlmap**
- Inputs: URL (with or without parameters), data (POST body), cookies, level/risk
- Guided mode: asks "Do you have a login session? Paste your cookie."
- Runs sqlmap with `--batch` + user-specified options
- Parses found injections and extracted data summaries

**2.5 XSS / Input Fuzzing**
- Uses `ffuf` with XSS wordlist against a parameterised URL
- Input: URL with FUZZ placeholder, wordlist

**2.6 HTTP Header Analysis**
- Uses `curl -I` + manual parsing
- Flags missing security headers: CSP, HSTS, X-Frame-Options, etc.
- Colour codes: present (green), missing (red), weak (yellow)

### MODULE 3: EXPLOITATION

**3.1 Metasploit Launcher**
- Scripty doesn't wrap msfconsole fully — instead it:
  - Asks: target IP, port, service
  - Suggests relevant modules from a curated list (common CVEs, EternalBlue, etc.)
  - Generates the `msfconsole -x "use ...; set RHOSTS ...; run"` command
  - Asks if user wants to launch it directly or copy to clipboard

**3.2 SearchSploit**
- Input: service name + version
- Runs `searchsploit` and displays results in a table
- Lets user select an exploit to read or copy

**3.3 Common Service Exploits (guided)**
- A sub-menu of known low-hanging-fruit attacks:
  - SMB: EternalBlue check, null session, anonymous share listing
  - FTP: Anonymous login test
  - SSH: Default credential test (small curated list)
  - Telnet: Banner grab + default cred test
  - HTTP: Default creds for common panels (router admin, Jenkins, etc.)

---

### MODULE 4: PASSWORD ATTACKS

**4.1 Hydra — Brute Force Login**
- Inputs: target IP, service (ssh / ftp / http-post-form / smb / rdp / etc.), username or list, password list
- For http-post-form: guided builder for login URL, POST body, failure string
- Shows command preview before running
- Streams output, highlights successful creds in green

**4.2 John the Ripper**
- Inputs: hash file path, wordlist, format (auto-detect or manual)
- Runs john, streams cracking progress
- Shows cracked passwords in a table

**4.3 Hashcat**
- Inputs: hash or hash file, attack mode (wordlist / brute / combinator / rule-based), hash type (with a searchable list of common types)
- Builds the hashcat command with appropriate flags
- Streams output

**4.4 Hash Identifier**
- Input: paste a hash
- Runs `hash-identifier` or uses Python `hashid` to identify likely hash type
- Shows top candidates

**4.5 Wordlist Manager**
- Lists available wordlists on the system (checks common Kali paths: `/usr/share/wordlists/`)
- Lets user set a default wordlist for other modules
- Option to download rockyou.txt if not present

---

### MODULE 5: NETWORK & TRAFFIC

**5.1 Netcat — Reverse Shell Listener**
- Input: port
- Runs `nc -lvnp <port>`
- Also generates reverse shell one-liners for bash, python, php, powershell — user can copy and paste to target

**5.2 Netcat — Port Check / Banner Grab**
- Input: host, port
- Runs `nc -zv` for check or connects and grabs banner
- Shows banner output

**5.3 Wireshark / tshark Capture**
- Lists available network interfaces
- Starts tshark capture with optional filter (input: BPF filter string or guided: HTTP / DNS / Credentials / All)
- Streams packet summary live
- Option to save to .pcap

**5.4 tcpdump Quick Capture**
- Same interface picker + filter builder
- Streams live

**5.5 ARP Scan / Local Network Discovery**
- Runs `arp-scan -l` or `netdiscover`
- Shows local hosts: IP | MAC | Vendor

**5.6 Traceroute**
- Runs `traceroute` / `tracepath`
- Shows hop table with latency

### MODULE 6: POST-EXPLOITATION

**6.1 LinPEAS / WinPEAS Launcher**
- Checks if linpeas.sh is downloaded, if not offers to fetch it
- Guides user through transfer method: python HTTP server, curl, wget, netcat
- Generates the command to run on target

**6.2 Linux Privilege Escalation Checklist**
- Interactive checklist of manual PrivEsc checks
- Each item: description + exact command to run on target + what to look for
- User can mark items as done / found / not applicable

**6.3 SUID/GUID Finder Commands**
- Generates and displays the `find` commands for SUID/SGID files
- Links findings to GTFOBins lookups

**6.4 Enum4linux — SMB Enumeration**
- Input: target IP
- Runs `enum4linux -a`
- Parses: shares, users, groups, password policy

**6.5 Persistence Techniques Reference**
- A reference panel listing common persistence methods (cron, rc.local, .bashrc, services, SSH keys)
- Each with example commands — copy-paste ready

## UX PATTERNS TO IMPLEMENT EVERYWHERE

1. **Command Preview**: Before running ANYTHING, show the exact command in a syntax-highlighted box and prompt "Run this? [Y/n]"

2. **Live Output Streaming**: Never swallow output. Stream subprocess stdout in real time inside a Rich Live panel.

3. **Plain English Summary**: After each tool runs, Scripty adds a "🧠 Scripty says:" block summarising what was found in simple terms.

4. **Next Steps Suggestions**: After each module, show "💡 What to do next:" with 2-3 contextual suggestions based on what was found.

5. **Save Results**: After each scan, offer to save raw output to `~/.scripty/results/<timestamp>_<module>.txt`

6. **Escape Hatch**: Pressing `q` or `Ctrl+C` anywhere returns to the previous menu gracefully, never crashes.

7. **Help Text**: Every prompt has visible placeholder/example text. No empty boxes.

8. **Sticky Target**: Once a target IP is entered, Scripty remembers it for the session and pre-fills it in subsequent prompts.

## IMPORTANT NOTES FOR THE VIBE CODER

- **This is for authorised testing only.** Add a disclaimer on first launch that the user must confirm before proceeding: "Scripty is for use on systems you own or have explicit written permission to test. Unauthorised access is illegal. Type AGREE to continue."

- **Don't overcomplicate the architecture.** The goal is a working, usable tool. Favour clarity over cleverness.

- **Test each module independently.** Each module file should be importable and runnable in isolation.

- **Handle missing tools gracefully.** If a module requires a tool that isn't installed, show a friendly message with the install command rather than crashing.

- **Prioritise the experience.** A slow but beautiful, guided experience beats a fast but confusing one. The whole point is that it feels like having a mentor next to you.

- **Kali Linux is the primary target OS.** Assume most tools are available via `apt`. Path assumptions should reflect Kali defaults.
